# Гайд-онбординг: match-экран для новичка

**Backlog:** `9d543af5-05af-430e-a215-7488c3f63612` — «Онбординг: первый чат менее 2 минут после регистрации»
**Дата:** 2026-06-04
**Статус:** дизайн утверждён, готов к плану реализации
**Репозитории:** `spirits_front` (основное) + `spirits_back` (флаг + эндпоинт)

## Контекст и проблема

Премиса задачи сместилась по данным (см. комментарий к задаче):
- `median_time_to_first_chat_min = 0.7` — скоростной барьер «первый чат < 2 мин» **уже выполняется**, это не проблема.
- Реальная проблема — **конверсия/match**: новичок после логина видит сетку из ~16 карточек ассистентов без контекста и должен выбрать сам. Большинство апрельских регистраций не дошли до 2-го дня.

**Цель:** при первом входе помочь новичку быстро найти «своего» специалиста и попасть в осмысленный первый чат — вместо холодного «выбери сам из 16».

Текущее состояние кода:
- Неавторизованный → `OnboardingPage` (это экран **логина**: phone/SMS/OAuth, не продуктовый онбординг).
- После логина → `/chat` → `ChatPage` → `ChatInterface` + `AssistantSelection` (сетка всех ассистентов, `visibleAssistants = assistants`, без группировки).
- Есть `preferred_agent` (авто-выбор, если задан) и `welcome_message` (общий, в ru пустой).
- Флага «прошёл онбординг» нет; гайд-флоу нет.

## Дизайн-решение

Направление — **гайд-флоу из одного экрана** (выбрано из вариантов: минимальный шаг / гайд-флоу / оба).

### Поток

1. Логин → `GET /webhook/profile` возвращает `onboarded: boolean`.
2. `onboarded === false` → вместо чата показываем **match-экран** «С чего начнём?» с 6 темами.
3. Клик по теме → выбирается привязанный ассистент → `POST /webhook/onboarding/complete` (ставит `onboarded=true`) → вход в чат. Ассистент встречает **обычным приветствием** (см. ниже), без контента под тему.
4. Внизу экрана — **«Показать всех специалистов»** → тоже `complete` (ставит флаг) → привычная сетка из 16 карточек.
5. `onboarded === true` → сразу обычный чат. Вернуться к подбору можно кнопкой **«Подобрать специалиста»** в шапке чата — она работает всегда, независимо от флага.

### Темы → ассистент

Привязка по стабильному `name` ассистента (per `spirits_back/CLAUDE.md` — `name` не меняется; `id` приведён справочно):

| Тема | Ассистент (`name`, id) | Примечание |
|---|---|---|
| 🧭 Разобраться в себе, эмоции | Оля (#2) | психолог; ядро для personal_growth-персоны |
| 📈 Цели и личный рост | Миша (#1) | коуч ICF |
| 💼 Карьера и работа | Ирина (#4) | HR / карьерный навигатор |
| 💰 Бизнес и деньги | Андрей (#7) | точка входа; **направляет** к Анне/Алексею/Виталию |
| 🔮 Практики самопознания | Райя (#14) | Human Design; рядом Шанкара/Лиана/Маша |
| 🤔 Пока не знаю / просто смотрю | Роман (#12) | координатор — задаёт пару вопросов и направляет (синергия с уже отгруженными suggested-prompt чипами Романа, backlog 8ebe8593) |

Копирайтинг: использовать глагол **«направляет»**, не «роутит» (анг­лицизм) — здесь и во всех пользовательских текстах.

### Приветствие после матча

Вариант **«обычное приветствие»** (без текста под тему). При входе в чат через матч ассистент показывает display-only вводное сообщение по единому шаблону:

> «Здравствуйте! Я {display_name}, {роль}. Чем могу помочь?»

`{роль}` берётся из существующей функции `getRoleForAssistant(description)` (ChatInterface). Один шаблон на всех 6 — отдельный контент под каждую тему **не пишем**. Сообщение display-only (как текущий `welcomeMessage`), в историю чата не сохраняется.

## Компоненты и изменения

### Бэкенд (`spirits_back`)

1. **Миграция** `src/.../migrations/NNN_onboarded_flag.sql`:
   - `ALTER TABLE ai_profiles_consolidated ADD COLUMN IF NOT EXISTS onboarded boolean DEFAULT false;`
   - **Бэкфилл** — все, у кого уже есть история чата, считаются прошедшими онбординг (экран не увидят):
     ```sql
     UPDATE ai_profiles_consolidated p
     SET onboarded = true
     WHERE EXISTS (
       SELECT 1 FROM custom_chat_history c
       WHERE c.session_id = p.user_id
          OR c.session_id LIKE p.user_id || '\_%'
     );
     ```
     Связка: `custom_chat_history.session_id` (text) — это телефон пользователя (= `ai_profiles_consolidated.user_id`, text), опционально с суффиксом `_<agentId>` (напр. `79656445804_4`). **Перед прогоном на проде сверить фактический формат `session_id`** одним SELECT'ом (формат — единственное допущение бэкфилла; при расхождении скорректировать предикат).
2. **`profile.service.ts`** (~строка 48): добавить `onboarded: row.onboarded === true` в объект ответа `GET /webhook/profile`.
3. **Новый эндпоинт** `POST /webhook/onboarding/complete` (под `JwtGuard`): `UPDATE ai_profiles_consolidated SET onboarded = true, updated_at = now() WHERE user_id = $1`. Идемпотентный, возвращает `{ onboarded: true }`. Разместить в profile-модуле (рядом с `profile-update`).

Зеркалит существующий паттерн `isadmin` (`001_core_schema.sql:847`) и `preferred_agent`.

### Фронтенд (`spirits_front`)

1. **`components/onboarding/OnboardingMatch.tsx`** — новый компонент match-экрана:
   - Заголовок «С чего начнём?», подзаголовок, 6 кнопок-тем из const `ONBOARDING_THEMES` (эмодзи, лейбл, `assistantName`).
   - Низ: ссылка «Показать всех специалистов →».
   - Пропсы: `assistants`, `onPickTheme(assistant)`, `onShowAll()`.
   - Мобайл-first, Tailwind, в стиле существующих карточек.
2. **`ONBOARDING_THEMES`** — массив `{ emoji, label, assistantName }` (привязка из таблицы выше). Матч ассистента: `assistants.find(a => a.name === theme.assistantName)`; если не найден → фолбэк на Романа.
3. **`ChatPage` / `ChatInterface`** — логика показа:
   - Если `user.onboarded === false` && ассистенты загружены && пользователь сам ещё не выбрал → рендерим `OnboardingMatch` вместо чата.
   - `onPickTheme(a)` → `setSelectedAssistant(a)` + вызвать `completeOnboarding()` + показать generic-приветствие.
   - `onShowAll()` → `completeOnboarding()` + показать обычную сетку `AssistantSelection`.
4. **Кнопка «Подобрать специалиста»** в шапке чата → открывает `OnboardingMatch` (через локальный стейт, без зависимости от флага). Доступна всегда.
5. **`services`**: `completeOnboarding()` → `apiClient.post('/webhook/onboarding/complete')`; локально выставляет `user.onboarded = true` (оптимистично).
6. **`AuthContext.User`** += `onboarded?: boolean`; маппинг поля из `GET /webhook/profile`.
7. **i18n**: ключи для заголовков, тем, кнопок, generic-приветствия в `ru.json`/`en.json`.

## Данные и состояние

- Источник истины — `ai_profiles_consolidated.onboarded` (БД). Фронт читает через `GET /webhook/profile`, пишет через `POST /webhook/onboarding/complete`.
- Один раз показанный и завершённый онбординг (`onboarded=true`) больше не появляется при последующих логинах. Повторный доступ — только явной кнопкой «Подобрать специалиста».

## Отказоустойчивость (края)

- **Профиль не загрузился / `onboarded` неизвестен** → fail-open: показываем обычный чат, не блокируем возвращающихся.
- **Привязанный ассистент отсутствует в ростере** → фолбэк на Романа (#12).
- **`completeOnboarding()` упал** → всё равно входим в чат; флаг до-проставится при следующем заходе (показ экрана повторно — приемлемая деградация, не блокер).
- **Существующие пользователи** не должны видеть экран → закрывается бэкфиллом миграции.

## Тестирование

**API smoke** (`tests/smoke/smoke.js` + при необходимости e2e):
- `GET /webhook/profile` отдаёт поле `onboarded`.
- `POST /webhook/onboarding/complete` ставит `onboarded=true`; повторный вызов идемпотентен (остаётся `true`).

**Playwright** (`tests/playwright/`):
- Новичок (`onboarded=false`, через тест-аккаунт со сброшенным флагом) видит match-экран; клик «🧭 Разобраться в себе» → чат с **Олей**, флаг выставлен.
- Возвращающийся (`onboarded=true`) → сразу чат, без match-экрана.
- «Показать всех специалистов» → сетка `AssistantSelection`; флаг выставлен.
- «Подобрать специалиста» в шапке → повторно открывает match-экран.

**Деплой:** только через `bash ~/Downloads/spirits_back/scripts/deploy.sh` (двухфазный test → smoke → prod → smoke).

## Вне scope (YAGNI)

- Контент-приветствия и suggested-prompts под каждую из 6 тем (выбран «обычное приветствие»).
- Многоэкранный wizard (3–4 экрана) — только один match-экран.
- Persona-detection / ML-матч — привязка статичная по теме.
- A/B-тест вариантов онбординга — отдельной задачей, если понадобится по данным.
- Аналитика-метрики онбординга (воронка match→первое сообщение) — желательна, но отдельным инкрементом; не блокирует MVP.
